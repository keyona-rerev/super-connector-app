## 1. BRAIN DUMP SYSTEM — PERMANENT INFRASTRUCTURE

### Problem
There was no reliable path for Claude to push brain dump data (initiatives, sub-projects, contacts) to Railway from a conversation. The GAS `gas_run_function` MCP doesn't have the execute OAuth scope. The Claude container can't reach Railway directly due to network allowlist.

### Solution
**`POST /brain-dump`** endpoint added to Railway API. Accepts a structured payload of initiatives (with nested sub_projects), contacts, and action_items. Handles all ID generation internally. Returns a full results summary. One call, no GitHub files, no cleanup.

Future brain dump sessions end with Claude calling this single endpoint — no manual steps.

### Also added: `serverBrainDump(payload)` to Network CRM GAS script
For sessions where the web app is a better execution path.

## 4. SEARCH — COMPLETELY FIXED

### Root causes
1. `/search` was using `embed_profile` (document mode) instead of `embed_query` (query mode) — Voyage AI degrades significantly when query mode isn't used
2. No simple text search existed — the only path was semantic vector search, which fails for name lookups
3. Front-end `crmInput` was calling the semantic endpoint on every keystroke

### What was built
**New Railway endpoint:** `GET /contacts/search?q=&limit=` — ILIKE substring match across `full_name`, `organization`, `title_role`, `notes`, `venture`, `how_we_met`, `source`. Name matches rank first. No AI, no embeddings, instant.

**New GIN index** on `contacts.profile` for fast text search.

**Fixed:** `POST /search` now uses `embed_query` (correct Voyage AI query mode).

**Fixed:** `POST /follow-ups/search` and `POST /content/search` also now use `embed_query`.

### Front-end behavior (config.js search patch)
- **Typing** → hits `GET /contacts/search?q=` (250ms debounce, text match, instant)
- **Enter** → same text search (names are not descriptive queries — semantic is not appropriate here)
- **Card clicks** → contacts stored in `_contactStore{}` keyed by ID, no more inline `JSON.stringify` in onclick
- **Badge** shows only on semantic results, not text results

## 6. HOW EXECUTION WORKS FROM CLAUDE SESSIONS

This was clarified and fixed permanently this session.

**The correct pattern:**
1. The web app at `keyona-rerev.github.io/super-connector-app` calls Railway directly via `fetch()` from the browser
2. GAS → Railway is the path for automated/background operations (T018, Phoebe, etc.)
3. Claude cannot call Railway from this container (network allowlist blocks it)
4. Claude CAN read/write GAS scripts and GitHub files directly via MCP

**For brain dumps:** Claude calls `POST /brain-dump` via a one-time `brain-dump-push.js` file committed to the GitHub Pages repo. It auto-fires when the app loads, pushes everything, then gets removed. Going forward this is replaced by calling `/brain-dump` directly from the web app interface.
